# Vault-by-Vault Release Rules

## Executive Summary

FUZE uses vault-by-vault release rules to define how each token allocation behaves before tokens enter circulation. FUZE does not treat every allocation as a simple unlock schedule, because each allocation has a different role, risk profile, release logic, access rule, vesting rule, stablecoin compensation relationship, and transparency requirement.

The vault-by-vault model supports FUZE v2’s Controlled Circulation Policy:

**Stablecoins pay for work. FUZE tokens align long-term participation.**

Each FUZE vault has one clear purpose, one release logic, one control model, and one public transparency rule. This gives investors, holders, contributors, users, and strategic partners a clearer understanding of how the fixed **500,000,000 FUZE** supply is governed.

Vault-by-vault release rules support tokenomics discipline, market trust, investor confidence, community clarity, anti-dump protection, stablecoin compensation separation, public vault transparency, and long-term ecosystem alignment.

## 1. FUZE Position

FUZE applies different release rules to different token allocation vaults.

FUZE does not use one universal unlock model across all allocations. Foundation tokens remain permanently locked. Migration tokens are reserved for eligible legacy-holder claims. Team and advisor allocations vest over time for long-term alignment. Holder incentives are earned through real usage and contribution. Liquidity tokens support market-structure needs. Treasury and selected ecosystem allocations can use policy-approved, market-aligned access windows where appropriate.

FUZE public principle:

**Not every allocation becomes public sale inventory. Each vault has a specific role and releases tokens only according to its purpose.**

FUZE uses the following vault release model:

| Vault / Allocation | Release Type | Public Buy Access | Core Rule |
|---|---|---:|---|
| Community Participation Allocation | Participation, product-user, contributor, reserve split | Yes, selected windows | Eligibility-controlled participation with vesting |
| BOARD / Surfboard Migration | Claim-based migration | No | Reserved for eligible legacy-holder claims |
| Team Vesting Vault | Long-term vesting + stablecoin compensation separation | Limited / exceptional | Team tokens align long-term builders, not short-term cash needs |
| Advisor Vesting Vault | Contribution-based vesting + milestone compensation | Limited / optional | Advisors earn long-term alignment through contribution |
| Foundation Vault | Permanent lock | No | Foundation principal never unlocks, sells, transfers, or enters circulation |
| Treasury Reserve Vault | Policy-based release / market-aligned access | Yes, selected windows | Strategic platform reserve, no scheduled unlock |
| Holder Incentives Vault | Earned rewards / capped campaigns | Usually no | Incentives are earned through real usage and contribution |
| Ecosystem Partnership Vault | Milestone-based release / strategic access | Sometimes | Partnerships are milestone-based or market-aligned |
| Liquidity Operations Vault | Controlled operational deployment | Usually no | Listing, liquidity, LP, exchange, and market-structure use only |
| Transparency / Stability Vault | Exceptional reserve use | Rarely | Trust/stability reserve, not normal sale inventory |

## 4. Investor and Community Relevance

Vault-by-vault release rules give investors and community members a clearer view of how FUZE manages token movement.

For investors, the model provides supply discipline. It shows that FUZE separates permanent locks, claim-based migration, long-term vesting, earned incentives, controlled liquidity deployment, treasury policy actions, and market-aligned access windows.

For holders, the model reduces confusion around unlock events. Each allocation has a defined purpose and a public reporting direction.

For community members, the model explains why community participation, product-user participation, contributor programs, and holder incentives follow different rules.

For team members, advisors, and contributors, the model separates stablecoin compensation from long-term FUZE token alignment.

For strategic partners, the model creates a clearer framework for milestone-based partnership release and market-aligned strategic participation.

For product users, the model supports the FUZE principle that product usage comes first and broader ecosystem participation follows platform readiness.

## 6. Risk Boundaries and Safeguards

FUZE does not guarantee token price, liquidity, listing, profit, return, payout, or market performance.

FUZE applies the following safeguards through vault-by-vault release rules:

| Risk Area | FUZE Safeguard |
|---|---|
| All allocations treated as sell inventory | Each vault has a specific purpose and release rule |
| Automatic unlock pressure | FUZE uses locks, vesting, claims, earned releases, and policy windows |
| Team or advisor sell-pressure concern | FUZE separates stablecoin compensation from token alignment |
| Migration misuse | BOARD / Surfboard Migration Vault is reserved for eligible legacy-holder claims |
| Foundation misuse | Foundation principal remains permanently locked |
| Treasury ambiguity | Treasury Reserve uses policy-based action, timelock, reporting, and selected access windows |
| Incentive abuse | Holder incentives use campaign caps, eligibility, anti-abuse controls, and delayed claims |
| Partner allocation misuse | Ecosystem Partnership Vault uses milestone-based or market-aligned release |
| Liquidity confusion | Liquidity Operations Vault supports market-structure needs, not normal public sale |
| Stability reserve misuse | Transparency / Stability Vault is reserved for exceptional trust or stability actions |
| Public access confusion | Public Vault Access is eligibility-controlled, lockup-aware, compliance-gated, and market-aligned |
| Token / credits / payout confusion | FUZE separates FUZE token, Platform Credits, and stablecoin payout systems |

FUZE does not guarantee exchange listing, listing timing, liquidity, price performance, or market outcome.

Vault-by-vault release rules do not remove market risk. They define how FUZE governs allocation purpose, release discipline, public reporting, access control, and long-term ecosystem alignment.
